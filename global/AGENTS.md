本文件是 Codex 全局协作规则，适用于所有项目。项目内若有 AGENTS.md、README、开发规范或用户当轮要求，必须一并遵守；发生冲突时，以更具体、更新的项目规则或用户要求为准。

## 总原则

1. 修改代码、配置、依赖、文档结构、远程仓库或本地状态前，必须先理解相关上下文。
2. 不允许在不了解项目结构、调用链、已有风格、目标来源和影响范围的情况下直接动手。
3. 每次交付都要能说明：改了什么、为什么这样改、如何验证。

## 执行前检查

涉及代码或项目文件变更时，先做最小必要调查：

- 确认当前目录、项目结构、相关文件、调用链和已有实现风格。
- 确认可复用的工具类、枚举、常量、异常体系、响应结构、组件、脚本或项目约定。
- 确认是否存在用户或其他线程的未提交变更，禁止覆盖、回滚或格式化非本人改动。
- 涉及安装、删除、同步、推送、远程仓库、第三方资源时，必须先确认准确来源、路径、版本、已有状态和影响范围；禁止按近似名称或猜测结果执行。

## 工具使用规则

- 涉及第三方库、框架、SDK、API 的新增、升级、配置或疑难用法时，优先使用 Context7 查询当前文档，不要只凭模型记忆。
- 涉及复杂 Bug、架构设计、迁移方案、多模块改造、技术选型或高风险决策时，优先使用 sequential-thinking 做结构化分析，再执行。
- 涉及公开网页资料采集、官网/帮助中心/博客/更新日志/竞品页面整理、网页转 Markdown/JSON、批量爬取公开站点内容，或 Context7 不覆盖的非标准文档站时，优先考虑使用 Firecrawl。
- Firecrawl 只用于公开网页和用户明确授权的内容；不要抓取登录后私密页面、受权限保护内容、敏感数据或用户未授权的数据源。
- 技术库、框架、SDK、API、CLI 的官方用法仍优先使用 Context7；需要登录态、用户已登录 Chrome 页面、真实浏览器视觉效果、复杂交互调试或表单操作时，使用 browser / Chrome / Playwright，不用 Firecrawl 替代。
- 使用 Firecrawl 时保持最小抓取范围：优先单 URL 或明确页面集合；批量 crawl 前先确认目标范围、输出格式和是否需要深度爬取。输出事实时保留来源 URL，并区分网页原文提取、整理归纳和模型推断。
- 普通小改动、明确的局部修复或无需外部资料的任务，不强制调用 MCP，避免增加不必要成本。

## Skill 调用规则

- `$coding-router` 是代码工程类 skill 路由器。当用户提出 Bug 修复、报错排查、测试/构建失败、新功能实现、重构、代码评审、Java/Vue 代码规范、第三方库/API/CLI 用法或完成前验证任务时，先使用它选择最小必要工程 workflow；它只负责路由，不直接实现代码。
- 代码工程任务不要默认同时调用 `root-cause-fixer`、`java-code-discipline`、`vue-code-discipline`、`implementation-planner`、`technical-solution-designer` 和 `superpowers` 全家桶；必须按 `$coding-router` 的判断选择 0-2 个核心 workflow。
- 如果任务主要是 UI、UX、前端视觉、Figma、动效或视觉验收，优先走 `$ai-design-router`；如果只是项目记忆、PRD 拆解、里程碑或多线程编排，走 `$project-memory-manager` 或 `$codex-project-orchestrator`，不要让 `$coding-router` 接管项目管理。
- `$ai-design-router` 是设计类 skill 路由器。当用户提出 UI、UX、前端视觉、页面/组件设计、重设计、Figma、设计系统、动效、视觉优化或视觉验收任务时，先使用它选择最小必要 skill 组合；它只负责路由，不直接实现 UI。
- 设计类任务不要默认同时调用 `frontend-design`、`taste-skill`、`ui-ux-pro-max`、TypeUI 候选和 Figma/GSAP 全家桶；必须按 `$ai-design-router` 的判断选择 1-2 个核心 skill，只有用户明确要求完整设计到代码闭环时才增加。
- `$humanizer` 是文本去 AI 味润色 skill，只在用户明确要求 humanize、改写、润色 Markdown/文案时使用；不要自动用于技术交付、Bug 根因、验证日志、项目记忆、代码解释或事实性文档。
- `$project-memory-manager` 是轻量项目日志 skill，用来回答和维护“这个项目做过什么、现在停在哪、下个会话从哪里继续”。当用户要求继续项目、记录本次工作、初始化/更新项目记忆、回顾项目进度，或一个项目明显会跨多个 Codex 会话持续推进时，必须显式使用它。
- `$project-memory-manager` 默认只维护 `docs/project-memory/00-current-status.md`、`01-activity-log.md`、`02-decisions.md`、`03-next-handoff.md`；不默认创建任务看板、ownership、inbox、worker 审核或复杂校验流程。
- `$codex-project-orchestrator` 是高级项目编排 skill。只有当用户明确需要 PRD 拆解、功能点清单、里程碑、任务看板、多线程/多 worker 协作、集成计划或主线程进度分析时，才显式使用它。
- 普通小改动、局部 Bug 修复、改文案、一次性问答、只读状态查询、提交推送，默认不调用这两个 skill；除非用户明确要求记录项目历史、继续长期项目或做项目编排。

## 需求拆分与项目推进

- 当用户提出灵感、模糊想法或未成形需求时，先做轻量整理：目标、当前理解、核心场景、直接下一步、待确认问题；不要默认创建完整任务看板或长路线图。
- 当用户给出正式 PRD、大功能、多模块目标，或明确要求开发规划/里程碑/任务拆分时，再做完整拆分；必要时使用 `$codex-project-orchestrator`。
- 完整拆分至少包括：目标、核心场景、功能点、模块边界、里程碑、可执行任务、验收标准和待确认问题。
- 需求不完整时，基于现有信息做最小可执行整理，并明确假设；不要因为没有正式 PRD 就停止推进。
- 如果用户明确要求立即实现某个小改动、修 Bug、改文案或局部调整，则不强制做完整需求拆分。

## 项目记忆与跨线程协作

- 长期项目、多个 Codex 会话连续推进、或用户要求“记录/继续/回顾项目”时，优先建立或读取仓库内 `docs/project-memory/`。
- `docs/project-memory/` 默认是轻量项目日志；`AGENTS.md` 只放长期规则，不记录当前进度、临时任务状态或会话事实。
- 新会话继续项目前，优先读取 `00-current-status.md`、`03-next-handoff.md`、`01-activity-log.md` 最近 3 条，再看本轮任务相关代码。
- 会话结束时，如果本轮产生了可复用项目事实，更新 `00-current-status.md`，并向 `01-activity-log.md` 追加本次做了什么、改了哪些文件、如何验证、还有什么没做、下次从哪里继续。
- 只有关键产品、技术、验收口径变化才写入 `02-decisions.md`；不要把普通实现细节、泛化建议或未确认路线图写成项目事实。
- 如果项目仍使用旧 `docs/project/` 或复杂版 `docs/project-memory/`，先作为历史上下文读取；只有用户明确要求迁移或覆盖时，才整理为轻量项目日志。

## Bug 处理

遇到 Bug 时必须优先定位根因：

- 先复现或收集证据，再判断原因。
- 从症状反向追踪到第一处错误数据、错误状态、错误边界或错误配置。
- 禁止用硬编码、吞异常、临时 if-else、绕过校验、默认值掩盖等方式只做表面修复。
- 如果无法证明根因，必须明确说明不确定点、已排除范围和下一步验证方式。

## 代码修改

- 优先复用项目已有结构、工具、模型、组件、异常处理、响应格式和编码风格。
- 保持职责单一、结构清晰，不新增无必要的抽象、设计模式、依赖、工具类或机制。
- 只修改完成任务所需的最小范围，不顺手重构、格式化、清理或重命名无关代码。
- 删除代码时只删除本次改动导致无用的内容；不要删除预先存在但无关的死代码，除非用户明确要求。
- 编写和修复的代码必须符合当前项目风格，而不是引入个人偏好的新风格。

## 语言与框架规范

- 使用项目已有语言、框架、目录结构和工程约定，不随意替换技术栈或引入新机制。
- Java 项目遵循阿里巴巴 Java 开发手册的基本规范。
- 复杂业务逻辑、核心判断、难理解代码块需要添加简洁中文注释。
- 不为显而易见的赋值、getter/setter、简单分支添加空洞注释。

## 验证要求

- 修改后必须运行与改动范围匹配的验证：单元测试、集成测试、构建、类型检查、lint、接口验证或必要的手动验证。
- 如果测试或验证无法运行，必须说明原因、已做的替代检查和剩余风险。
- 不允许把“代码已写”描述为“功能已完成”；只有实现、验证、必要文档同步都完成后，才能称为完成。
- 涉及真实配置、短信、地图、支付、医疗数据、外部服务等能力时，不能用 mock 或假数据冒充真实接入完成。

## 回复要求

- 回复保持简洁，只说和交付有关的内容。
- 默认按“改了什么、为什么这样改、如何验证、剩余问题”汇报。
- 不输出无关长文档，不主动扩展大量退回策略或兜底方案。
- 如果用户明确要求先分析、评审或规划，不要直接改代码。
- 如果需求存在高风险歧义，先提出具体问题；不要沉默猜测后执行。
