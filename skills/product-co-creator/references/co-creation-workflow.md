# Co-Creation Workflow

Use this workflow when the user already has a product direction and wants help shaping it into something buildable.

## 1. Anchor the idea

Clarify:

- What kind of product it is
- Which platform it targets
- Who it serves first
- What job the user is hiring it to do
- What successful use looks like

If the user gives only a noun such as "健身记录小程序", do not rush into features. First infer the core job:

- Record
- Plan
- Guide
- Motivate
- Analyze
- Socialize

## 2. Select the right role stack

Do not answer as a generic PM only. Add the most relevant expert perspectives for the domain.

Examples:

- Fitness: product manager + health coach + senior trainer
- Education: product manager + curriculum designer + learner success lead
- SaaS tooling: product manager + workflow designer + operations lead
- Consumer content: product manager + creator strategist + growth lead

Use the domain experts to shape:

- terminology
- trust boundaries
- guidance logic
- progression design
- user motivation

## 3. Write the product promise

State the product in one sentence:

`This product helps <target user> achieve <desired outcome> through <core mechanism>.`

Then define:

- The user group to serve first
- The highest-frequency scenario
- The most painful unmet need
- The product's differentiating principle

## 4. Design around 3 to 5 core scenarios

Avoid feature soup. Build around a few scenarios that are frequent, valuable, and behavior-shaping.

For each scenario, define:

- Trigger
- User goal
- Required input
- Product response
- Success signal

## 5. Cut the MVP aggressively

The MVP should prove the product promise, not include every good idea.

Keep in v1:

- actions required for the core scenario
- minimum data needed to personalize or track value
- a basic loop that brings the user back

Move to later phases:

- advanced analytics
- deep personalization
- social systems
- heavy automation
- large content libraries

## 6. Add structural detail

After the MVP is clear, specify:

- key modules
- screen or page structure
- navigation logic
- data objects
- content units
- rules and states

This is the point where the product becomes buildable instead of inspirational.

## 7. Finish with operating detail

Always end with:

- success metrics
- retention hooks
- version roadmap
- open assumptions
- what should be validated in the first release
