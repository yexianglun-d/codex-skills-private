# Output Framework

Use one of these output patterns based on how detailed the user wants the response to be.

## Pattern A: Quick Product Direction

Use when the user asks "这个产品该怎么做" but does not want a full spec yet.

Output:

1. Product definition
2. Target users
3. Core scenarios
4. MVP modules
5. Key differentiation
6. First-version metrics

## Pattern B: Product Blueprint

Use when the user wants a complete co-created plan.

Output:

1. Product positioning
2. User segments
3. User problems and goals
4. Core use cases
5. MVP scope
6. Feature module breakdown
7. User flow
8. Data or content structure
9. Retention loop
10. Metrics
11. Version roadmap
12. Open questions

## Pattern C: Delivery-Ready Requirement Pack

Use when the user wants material that can be handed to design or engineering.

Output:

1. Product goal
2. Scope and non-scope
3. User stories
4. Functional requirements
5. Core states and edge cases
6. Data fields and rules
7. Interaction notes
8. Acceptance criteria
9. Post-launch metric checks

## Quality Bar

A strong output should:

- stay specific to the product's real domain
- make clear tradeoffs
- avoid bloated MVPs
- connect features back to user value
- be structured enough for execution, not just ideation
