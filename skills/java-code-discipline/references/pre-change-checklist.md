# 修改前检查清单

用于任何 Java 代码、配置、依赖或项目文件变更。

## 必查上下文

- 当前目录、模块结构、相关包路径和入口文件。
- 目标功能的调用链：Controller / API -> Service -> Repository / Mapper -> 外部服务或数据库。
- 相关数据流：请求参数、DTO、Entity、枚举、状态字段、返回结构、异常路径。
- 已有实现风格：命名、分层、事务、日志、异常、测试、注释、配置读取方式。
- 可复用内容：工具类、常量、枚举、异常类型、响应包装、校验器、Mapper 方法、脚本。
- 当前工作区状态：是否有未提交改动，哪些文件可能是用户或其他线程改过的。

## 修改原则

- 先复用项目已有模式，再考虑新增代码。
- 新增类、工具、抽象、依赖前，必须能说明它解决了真实复杂度或明确复用点。
- 只修改当前目标需要的最小范围，不做无关格式化和顺手清理。
- 删除代码只删除本次改动导致无用的内容；预先存在但无关的死代码不要动。
- 复杂业务逻辑、核心判断、容易误读的边界条件添加简洁中文注释。
- 不给显而易见的赋值、getter/setter、简单 if 分支添加空洞注释。

## Java 基础约束

- 类名使用 UpperCamelCase；方法、参数、局部变量使用 lowerCamelCase；常量使用 UPPER_UNDERSCORE。
- 集合、数组、批量结果命名要表达复数或集合语义，避免 `data`、`list1` 这类弱含义命名。
- 不使用魔法值；业务状态、错误码、配置项优先走已有枚举、常量或配置。
- 对外接口、数据库字段、序列化字段命名变更要谨慎，先确认兼容影响。
- 金额、精确小数计算避免使用 `double` / `float` 承载业务精度。
- 日志不输出密码、token、身份证、手机号、医疗隐私等敏感信息。

## 开始修改前的最低结论

动手前至少能回答：

- 要改的是哪个边界的问题？
- 已有系统里同类逻辑怎么写？
- 这次改动会影响哪些调用方或数据状态？
- 修改后用什么验证它真的生效？
