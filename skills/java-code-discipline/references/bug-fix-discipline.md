# Bug 修复纪律

用于崩溃、测试失败、接口异常、数据错误、状态不一致、线上回归等问题。

## 根因定位流程

1. 先复现或收集证据：错误日志、请求参数、返回值、数据库状态、测试输出、浏览器或接口现象。
2. 从症状反向追踪到第一处错误：错误数据、错误状态、错误边界、错误配置、错误并发假设或错误外部依赖。
3. 写出根因判断：`因为 X 在 Y 条件下产生 Z，所以导致当前现象`。
4. 在根因所在边界修复，而不是在下游用兼容逻辑掩盖。
5. 补充或运行能覆盖该根因的验证，必要时覆盖相邻场景和回归场景。

## 禁止的表面修复

- 用硬编码特殊值绕过问题。
- 捕获异常后不处理、不记录关键上下文，或直接吞掉。
- 加临时 if-else 只让当前输入通过。
- 跳过校验、跳过权限、跳过事务、跳过真实外部服务。
- 用默认值掩盖丢失数据、错误配置或错误状态。
- 改调用方绕开问题，但不修正源头错误。

## 修复质量检查

- 修复位置是否靠近真正的状态写入、数据转换、边界校验或配置来源？
- 是否保持原有异常体系、响应结构、日志风格和事务边界？
- 是否避免扩大影响范围？
- 是否有测试或手动证据证明修复前失败、修复后通过？
- 是否说明无法完全证明的部分和下一步验证方式？

## 回复时必须讲清

- 根因是什么。
- 修复点为什么选在这里。
- 验证覆盖了什么。
- 还有哪些无法验证或需要用户提供环境的部分。
