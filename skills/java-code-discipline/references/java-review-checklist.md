# Java 评审清单

用于代码评审、重构前自查、提交前检查。

## 分层与职责

- Controller 只处理协议、参数校验、认证上下文和响应组装，不承载复杂业务。
- Service 负责业务规则、事务边界和跨组件协作。
- Mapper / Repository 只做数据访问，不塞业务判断。
- DTO、VO、Entity 的职责清晰，不混用导致接口字段和持久化字段互相污染。
- 外部服务调用有清晰边界、超时、错误处理和必要日志。

## 异常与日志

- 异常类型、错误码、响应结构优先复用项目已有体系。
- 不使用 `printStackTrace()`；日志要包含排障所需上下文，但不泄露敏感信息。
- 不在低层随意吞异常；需要转换异常时保留原始 cause 或关键上下文。
- 日志级别要匹配严重程度，避免把正常业务分支打成 error。

## 数据与集合

- 空值处理要符合项目风格：边界校验、显式 Optional、空集合返回或异常，不混乱使用。
- 集合遍历、分页、批量查询注意 N+1、重复查询和大集合内存风险。
- 字符串比较、枚举比较、时间比较、金额计算要避免隐式精度和空指针问题。
- 时间字段明确时区、格式和业务含义，避免把展示时间和存储时间混在一起。

## 事务与并发

- 事务边界放在业务一致性所在的 Service 层，避免过大或过小。
- 事务内避免不必要的远程调用、长耗时操作和不可控阻塞。
- 并发更新需要确认锁、版本号、唯一约束或幂等机制。
- 定时任务、消息消费、回调处理要考虑重复执行和乱序到达。

## 数据库与 MyBatis

- SQL 条件、排序、分页和关联查询要确认索引和数据量影响。
- 动态 SQL 避免拼接注入风险，参数绑定优先。
- 字段新增、删除、含义变化要同步实体、Mapper、迁移脚本、接口文档或测试。
- 批量操作要考虑事务、失败语义和重复提交。

## 测试与交付

- 高风险业务规则至少有单元测试或集成测试覆盖。
- Bug 修复测试要覆盖触发根因的输入，不只测正常路径。
- 无法运行测试时，说明环境原因、替代验证和剩余风险。
- 不把代码改完等同于功能完成；需要验证和必要文档同步。
