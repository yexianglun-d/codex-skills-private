# 项目风格贴合清单

用于确保 Java 修改符合当前系统，而不是引入个人偏好的新写法。

## 先找同类实现

优先搜索并阅读：

- 同模块、同业务域、同接口层级的已有实现。
- 已有异常、错误码、响应包装、权限校验、参数校验、日志模板。
- 已有枚举、常量、配置类、工具类、Mapper XML 或注解风格。
- 已有测试写法、测试数据组织、构建命令和本地启动方式。

## 保持局部一致

- 包名、类名、方法名、DTO / VO / Entity 后缀遵循项目习惯。
- Controller、Service、Mapper 的职责分配遵循项目原有边界。
- 同类接口的返回结构、状态码、错误消息、分页结构保持一致。
- Mapper XML、注解 SQL、Query Wrapper 等数据访问风格不混用，除非项目已有这种混用规则。
- 配置读取方式遵循项目习惯，不随意改成新配置中心、新前缀或硬编码。

## 新增内容判断

新增类、方法、配置、依赖前确认：

- 是否已有同类能力可复用？
- 是否只为当前一个分支服务？如果是，是否直接内聚在当前类更合适？
- 是否改变了公共 API、数据库结构、序列化字段或外部协议？
- 是否需要同步测试、文档、迁移脚本或部署配置？

## 多线程协作保护

- 修改前查看工作区状态，区分自己的改动和用户/其他线程的改动。
- 不回滚、不覆盖、不格式化无关文件。
- 如果必须改到已有脏文件，先理解现有差异，再在其基础上追加最小修改。
- 发现其他线程正在改同一边界时，优先收敛到更小文件范围，并在交付中说明冲突风险。

## 完成判定

只有同时满足以下条件才称为完成：

- 代码、配置或文档改动已落地。
- 与改动范围匹配的验证已运行或明确说明无法运行原因。
- 没有引入无关差异。
- 交付说明能解释改动原因和验证结果。
