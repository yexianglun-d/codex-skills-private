# Vue Bug 修复纪律

用于页面异常、交互错误、渲染错乱、状态不同步、接口数据异常、构建失败、测试失败、线上回归等问题。

## 根因定位流程

1. 先复现或收集证据：浏览器现象、控制台错误、网络请求、组件状态、store 状态、路由参数、测试输出或构建日志。
2. 从症状反向追踪到第一处错误：错误 props、错误响应数据、错误 computed/watch、错误 store mutation/action、错误路由边界、错误样式覆盖或错误第三方实例生命周期。
3. 写出根因判断：`因为 X 在 Y 条件下产生 Z，所以导致当前现象`。
4. 在根因所在边界修复，而不是在下游通过默认值、定时器、强刷或额外 if 分支掩盖。
5. 补充或运行能覆盖该根因的验证，必要时覆盖 loading、empty、error、重复点击、快速切换和权限边界。

## 禁止的表面修复

- 用硬编码 id、固定索引、固定路由、固定尺寸绕过问题。
- 捕获接口或渲染异常后直接吞掉，或只打印无上下文日志。
- 用 `setTimeout`、`nextTick`、强制刷新页面掩盖生命周期或状态同步问题。
- 给异常数据加默认值后不追查数据来源、接口协议或状态写入源头。
- 绕过表单校验、权限判断、路由守卫、store 约束或项目封装。
- 为当前问题新建全局状态、事件总线或临时缓存，但不修正源头。

## 修复质量检查

- 修复位置是否靠近真正的状态写入、数据转换、响应处理、组件边界或样式来源？
- 是否保持原有 API 封装、错误提示、loading、empty、权限和路由风格？
- 是否避免扩大影响范围？
- 是否覆盖了触发 Bug 的原始路径和相邻回归路径？
- 是否说明无法完全证明的部分和下一步验证方式？

## 回复时必须讲清

- 根因是什么。
- 修复点为什么选在这里。
- 验证覆盖了什么。
- 还有哪些无法验证或需要用户提供环境的部分。
