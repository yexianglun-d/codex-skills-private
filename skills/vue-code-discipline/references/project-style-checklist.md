# 项目风格贴合清单

用于确保 Vue 修改符合当前系统，而不是引入个人偏好的新写法。

## 先找同类实现

优先搜索并阅读：

- 同页面、同模块、同业务域、同组件层级的已有实现。
- 现有 API 请求封装、错误处理、loading、toast/message、权限校验、表单校验和路由跳转方式。
- 现有 store/composable 命名、状态组织、缓存策略和数据转换位置。
- 现有基础组件、业务组件、样式变量、CSS class 命名和响应式布局方式。
- 现有目录结构：`views/pages`、`components`、`stores`、`api/services`、`router`、`utils`、`styles`、`types` 是否已有明确边界。
- 现有测试写法、构建命令、本地启动方式和浏览器验证方式。

## 保持局部一致

- 不在 Composition API 项目里新增 Options API 组件，除非局部文件本身已采用该风格。
- 不在 Vue 2 项目里擅自引入 Vue 3 专属写法或依赖。
- 不随意从 Vuex 迁移到 Pinia，或从 Pinia 退回临时全局变量。
- 不绕过项目已有 API client、错误提示、权限、埋点、路由守卫和环境变量读取方式。
- 不混用新的 CSS 方案或 UI 组件库，除非用户明确要求。
- 不在已有稳定结构中突然新增另一套 `modules`、`features`、`services` 或 `hooks` 目录命名，除非这是一次明确的结构治理。

## 企业级目录落位

- 新项目默认按 `enterprise-directory-structure.md` 建立分层目录。
- 新增页面放到业务域目录下，页面私有组件、私有 composable、私有类型优先内聚在该业务域内。
- 多业务复用组件放 `components/business`；纯 UI 基础组件放 `components/base`。
- 跨页面复用逻辑放 `composables`，跨系统工具放 `utils`，不要混放。
- API 请求按业务域拆分到 `api`，不要直接散落在组件里。
- 全局状态按业务域拆分到 `stores/modules` 或项目已有等价结构。

## 新增内容判断

新增组件、composable、store 字段、API 方法、类型、配置或依赖前确认：

- 是否已有同类能力可复用？
- 是否只为当前一个分支服务？如果是，是否内聚在当前组件更合适？
- 是否改变了公共组件 API、路由协议、接口参数、埋点字段、缓存键或外部行为？
- 是否需要同步测试、文档、mock、类型声明、环境配置或部署配置？

## 多线程协作保护

- 修改前查看工作区状态，区分自己的改动和用户/其他线程的改动。
- 不回滚、不覆盖、不格式化无关文件。
- 如果必须改到已有脏文件，先理解现有差异，再在其基础上追加最小修改。
- 发现其他线程正在改同一组件、store 或接口边界时，优先收敛到更小文件范围，并在交付中说明冲突风险。

## 完成判定

只有同时满足以下条件才称为完成：

- 代码、配置或文档改动已落地。
- 与改动范围匹配的验证已运行或明确说明无法运行原因。
- 没有引入无关差异。
- 交付说明能解释改动原因和验证结果。
