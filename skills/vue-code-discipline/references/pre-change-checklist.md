# 修改前检查清单

用于任何 Vue 代码、配置、依赖或项目文件变更。

## 必查上下文

- 当前目录结构、应用入口、路由配置、页面组件、相关子组件和组件调用关系。
- 目标功能的数据流：route/query/props -> store/composable/API -> 本地状态 -> template -> emit/路由跳转。
- 已有实现风格：Composition API / Options API、`<script setup>`、TypeScript、状态管理、请求封装、错误提示、表单校验、样式方案。
- 可复用内容：基础组件、业务组件、composable、store action/getter、API client、类型、常量、校验器、工具函数、样式变量。
- 构建与验证命令：`lint`、`type-check`、`test`、`build`、本地 dev server、浏览器验证方式。
- 当前工作区状态：是否有未提交改动，哪些文件可能是用户或其他线程改过的。

## 目录结构检查

- 判断项目是新项目、已有项目新增模块、还是历史结构治理。
- 新项目或新增模块无明确位置时，按 `enterprise-directory-structure.md` 放置文件。
- 已有项目存在稳定目录风格时，先遵循现状；不要为了套模板移动大量历史文件。
- 如果用户明确要求“企业级目录规范”“整理目录结构”“重构前端结构”，先给迁移边界和影响范围，再执行最小可验证迁移。

## 修改原则

- 先复用项目已有模式，再考虑新增组件、composable、store 字段或工具函数。
- 新增抽象前必须能说明它降低了真实复杂度，或会被多个同类场景复用。
- 只修改当前目标需要的最小范围，不做无关格式化、迁移和样式重写。
- 删除代码只删除本次改动导致无用的内容；预先存在但无关的死代码不要动。
- 复杂交互、核心状态转换、竞态处理或难理解边界添加简洁中文注释。
- 不给显而易见的赋值、简单 computed、简单事件转发添加空洞注释。

## Vue 基础约束

- 组件名使用有语义的 PascalCase；事件名、插槽名、CSS class 命名遵循项目既有风格。
- `props` 和 `emits` 必须清晰声明；TypeScript 项目优先复用已有类型。
- 不直接修改 `props`；需要可编辑副本时显式创建本地状态并处理同步边界。
- `v-for` 必须有稳定 `key`，不要用随机数或不稳定索引掩盖刷新问题。
- 避免在同一节点同时写 `v-if` 和 `v-for`，优先用 computed 预过滤数据。
- `watch`、`watchEffect`、生命周期钩子和异步副作用要明确触发条件与清理边界。
- DOM 直操作、定时器、全局事件监听、第三方实例必须有卸载清理。
- 组件内不堆积复杂业务流程；复杂逻辑按项目习惯拆到 composable、store、service 或工具函数。

## 开始修改前的最低结论

动手前至少能回答：

- 要改的是组件树、数据流、状态管理、API 边界、路由边界还是样式边界？
- 新增或移动的文件应该放在哪一层目录，依据是项目现状还是企业级默认结构？
- 已有系统里同类交互怎么写？
- 这次改动会影响哪些页面、组件、store 或接口调用？
- 修改后用什么验证它真的生效？
